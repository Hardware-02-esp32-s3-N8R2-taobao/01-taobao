# ESP32-S3 人脸坐标输出前置说明

## 结论

是的，这一步应该单独先跑通，而且它就是 `07` 方案的人脸追踪部分的前置条件。

原因很简单：

`07` 真正要驱动舵机追踪，核心输入不是“识别到了谁”，而是：

- 当前画面里有没有人脸
- 人脸框的 `x / y / w / h`
- 人脸中心点相对画面中心的偏差

只要这一步稳定，后面的舵机控制就只是控制算法问题。

## 为什么先做“坐标输出”

### 1. 先把视觉链路单独验证掉

先确认下面这些是否都正常：

- 摄像头初始化
- 图像采集
- 人脸检测模型运行
- 检测框输出
- 串口或日志打印坐标

### 2. 避免舵机问题干扰判断

如果一开始就把舵机也接上，会同时引入：

- 供电干扰
- 抖动
- PWM 配置问题
- 云台机械结构偏差

这样很难判断是“视觉没跑通”，还是“舵机控制有问题”。

### 3. 坐标输出本身就足够形成里程碑

只要你能稳定输出：

- 左上角坐标 `x1, y1`
- 右下角坐标 `x2, y2`
- 中心点 `cx, cy`

那后面就可以自然推进到：

1. 水平单舵机跟踪
2. 双舵机 `pan + tilt`
3. 再考虑识别具体是谁

## 这一阶段的最小目标

建议目标不要定得太大，先做到下面这 4 件事：

1. `ESP32-S3` 正常采集摄像头画面
2. 检测到人脸
3. 每帧输出人脸框坐标
4. 串口日志可稳定观察坐标变化

## 这一阶段的输出格式建议

建议直接统一成这种格式：

```text
face,score=0.92,x=86,y=42,w=74,h=74,cx=123,cy=79
```

这样后面你接：

- 串口上位机
- VS Code 串口监视器
- Python 可视化脚本
- 舵机控制逻辑

都会更省事。

## 这一阶段做完以后，下一步是什么

这一步跑通之后，再进入 `07`：

- 人脸识别
- 多目标筛选
- 舵机追踪
- 云台控制

也就是说，`08` 不是和 `07` 平行，而是 `07` 的前置阶段。
