# 固件升级方案

## 目标

本项目现在统一使用同一份 `.bin` 作为升级入口，但会根据实际场景自动拆分成不同升级方式：

- 空板、刚 SMT 完的板子、没有我们自定义 OTA 协议的板子：走串口全烧录。
- 已运行本项目固件、串口可通信的板子：走串口 OTA。
- 已联网、通过云端网页发起升级的板子：走云端 OTA。
- 历史遗留的旧版纯 `app.bin`：仍可 OTA，但不能空板全烧录。

核心原则只有一个：

- 用户始终选同一份升级包 `.bin`。
- 桌面端、服务端、网页端根据场景，从同一份包里提取自己需要的分段。

## 统一升级包格式

当前发布的主固件文件已经不是单一 app 镜像，而是一个“统一升级包”，文件扩展名仍然保持 `.bin`。

包格式特征：

- Magic：`YDFWPKG1`
- Header：`magic + manifest length`
- Manifest：JSON，描述版本、芯片、flash 参数、OTA 目标段、各段偏移与大小
- Payload：实际二进制分段内容顺序拼接

当前默认包含这些分段：

- `bootloader`
- `partition-table`
- `otadata`
- `app`

其中：

- 串口全烧录会使用 `bootloader + partition-table + otadata + app`
- 串口 OTA 只会提取 `app`
- 云端 OTA 也只会提取 `app`

这样就实现了“同一个文件，不同场景自动拆分”。

## 当前分区关系

ESP32-C3 固件当前是 OTA 双分区布局：

- `otadata @ 0xD000`
- `ota_0 @ 0x10000`
- `ota_1 @ 0x200000`

因此：

- 本地串口 OTA、云端 OTA，下发的都是纯 `app` 镜像
- 空板全烧录时，需要把启动相关段和 app 一起写进去

## 各场景升级策略

### Case A：空板 / 新 SMT 板 / 不带我们 OTA 协议的板子

处理方式：

- 在上位机“固件升级”页选择统一升级包
- 如果串口侧没有识别到本项目运行态协议，则自动策略切到“全烧录”
- 上位机会从统一升级包中提取 `bootloader / partition-table / otadata / app`
- 通过 `esptool` 按地址全烧录

适用对象：

- 刚出厂、刚焊好的空板
- 之前烧了其他程序的板子
- 板子里没有本项目串口 OTA 协议

特点：

- 不依赖设备当前是否有网络能力
- 不依赖设备当前是否已经跑起来
- 只依赖串口能够进入下载烧录流程

### Case B：板子已运行本项目固件，且串口连着电脑

处理方式：

- 上位机识别到设备已经运行本项目固件
- 自动策略优先走“串口 OTA”
- 上位机从统一升级包中只提取 `app`
- 沿用原来的串口 OTA 协议，给设备发送纯 app 镜像

特点：

- 速度比全烧录更合适
- 不动 bootloader / partition / otadata
- 适合日常开发调试升级

### Case C：板子已联网，从云端网页发起 OTA

处理方式：

- 网页端上传的仍然是统一升级包
- 服务端保存整包
- 当设备发起 OTA 下载时，服务端只从包中提取 `app` 段返回给设备
- 设备固件侧仍按原来的 HTTP OTA 逻辑处理纯 app 镜像

前提：

- 这种方式默认板子已经具备正常启动、联网和 OTA 能力
- 也就是它本身已经有 bootloader 和本项目固件运行基础

特点：

- 网页端和桌面端共用同一份发布包
- 设备侧 OTA 协议不用为了“整包格式”改造
- 服务端负责做“整包转 app 段”的拆包

### Case D：用户选择的是旧版纯 app bin

处理方式：

- 系统会把它识别成 `raw-app-bin`
- 允许串口 OTA
- 允许云端 OTA
- 不允许空板全烧录

原因：

- 旧版文件只包含 `app`
- 不包含 `bootloader / partition-table / otadata`
- 因此无法恢复一块空板

## 上位机自动判断逻辑

桌面端“固件升级”页的自动策略如下：

1. 如果识别到串口设备正在运行本项目固件，则走串口 OTA。
2. 如果串口已连接但没有识别到本项目 OTA 协议，则走全烧录。
3. 如果串口刚连接、还没收到任何识别信息，为避免误判，先等待识别结果，不立即升级。
4. 如果当前包只包含 `app`，则只允许 OTA，不允许全烧录。

这套逻辑的目的是：

- 对已经在跑我们固件的板子，优先轻量升级
- 对空板或非项目固件板子，自动回退到 IDF/esptool 的全烧录模式
- 避免用户维护两份不同的升级文件

## 服务端与网页端策略

服务端现在保存的是“统一升级包整包”，并额外记录：

- `packageFormat`
- `schema`
- `chip`
- `flashSettings`
- `otaSegmentRole`
- `supportsFullFlash`
- `segments`
- `otaSize`
- `otaSha256`

网页管理端现在会明确展示：

- 当前选中的固件是统一升级包还是旧版 app bin
- 是否支持空板全烧录
- 当前包里有哪些分段
- 云端 OTA 实际只会向设备下发 `app` 段

## 固件侧策略

设备固件侧串口 OTA / HTTP OTA 协议尽量保持不变：

- 设备看到的仍然是纯 `app` 镜像
- 不让设备直接理解统一升级包格式
- 包解析与拆段工作放在桌面端和服务端完成

这样可以降低设备侧风险，也更容易保持兼容。

## 发布建议

日常发布建议使用以下流程：

1. 运行 `01-build_firmware.py`
2. 产出原始 app 版本包和统一升级包
3. 发布时优先发统一升级包 `yd-c3_v<version>.bin`
4. 桌面端、网页端都使用这同一份文件

## 结论

现在的升级体系可以概括为一句话：

- 同一份 `.bin`，空板走全烧录，运行态设备走 OTA，网页 OTA 也复用同一份包，只是服务端自动提取 app 段。

这样后续维护时，发布文件、操作流程、用户理解成本都会明显更低。
