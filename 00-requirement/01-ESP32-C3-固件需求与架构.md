# ESP32-C3 固件需求与架构

## 1. 目标定位

本固件运行在 `ESP32-C3 SuperMini` 上，承担“传感器采集 + 网络上报 + 本地调试输出”的职责，是整个系统的数据入口节点。

当前仓库中已经存在可参考实现：

- `01-firmware/00-main-c3-firmware`

该固件当前主要完成：

- 连接 Wi-Fi
- 采集 `DHT11` 温湿度
- 通过 `MQTT` 向服务器周期上报
- 通过串口输出状态日志，供本地调试与 App 监控使用

当前主工程路径：

- `01-firmware/00-main-c3-firmware`

## 2. 固件需求

### 2.1 核心功能需求

1. 上电后自动初始化硬件与软件模块。
2. 自动连接预设 Wi-Fi。
3. 在 Wi-Fi 可用后自动连接 MQTT 服务。
4. 周期性采集传感器数据，当前至少支持 `DHT11`。
5. 将采集结果按统一 JSON 格式发布到指定 Topic，消息内容应以设备身份、固件版本、成功采集到的传感器列表为核心。
6. 当网络异常时，固件能够自动重连并恢复上报。
7. 串口持续输出关键状态，便于开发阶段诊断。

### 2.2 可扩展需求

1. 后续可扩展更多传感器，例如 `DS18B20`、`BMP280`、`BH1750`。
2. 后续可增加下行控制能力，例如水泵控制、参数更新、设备重启。
3. 后续可增加设备身份、版本号、时间戳、自检结果等元数据。
4. 后续可增加 OTA 升级能力。

### 2.3 稳定性需求

1. Wi-Fi 断开后应自动重连。
2. MQTT 断开后应自动重连。
3. 传感器读取失败时不能导致主流程卡死。
4. 当配置了多个传感器但只有部分传感器读取成功时，固件只上报成功读取到的数据，不得上传无效占位数据。
5. 当传感器配置存在但硬件未接好或读取失败时，串口日志应持续提示失败状态，便于 App 明确提醒用户排查接线或关闭该传感器配置。
6. 发布失败时应保留日志，便于定位问题。
7. 固件主循环应保持轻量，避免阻塞。

## 3. 当前建议架构

建议采用“分层 + 服务化”结构：

### 3.1 硬件驱动层

职责：

- 传感器驱动
- GPIO 管理
- OLED 或其他外设驱动

当前对应模块可参考：

- `main/dht11/dht11_sensor.c`
- `main/oled/oled_ssd1306.c`

### 3.2 设备配置层

职责：

- Wi-Fi 参数
- MQTT 参数
- Topic 常量
- 设备 ID、别名、采样周期

当前对应模块可参考：

- `main/include/app_config.h`
- `main/app/device_profile.c`

当前已落地的关键配置包括：

- `APP_WIFI_SSID`
- `APP_WIFI_PASSWORD`
- `APP_MQTT_URI`
- `APP_MQTT_USERNAME`
- `APP_MQTT_PASSWORD`
- `APP_MQTT_TOPIC_PREFIX`
- `APP_DEVICE_ID`
- `APP_DEVICE_ALIAS`
- `APP_FIRMWARE_VERSION`
- `APP_PUBLISH_INTERVAL_MS`
- `APP_DHT11_GPIO`

当前 MQTT 入口实际值：

- `APP_MQTT_URI = mqtt://117.72.55.63:1884`
- 当前真实生效的是 `MQTT TCP 1884`
- 不再使用旧的 `ws://117.72.55.63/mqtt`

### 3.3 网络服务层

职责：

- Wi-Fi 连接管理
- MQTT 连接管理
- 发布与订阅封装
- 连接状态同步

当前对应模块可参考：

- `main/network/network_service.c`

当前已确认的硬件变体联网策略：

- `supermini`：沿用普通 `ESP32-C3 SuperMini` 的直接 STA 连接逻辑
- `oled-screen`：先扫描附近 AP，再按已保存 SSID 选择可见热点，并锁定 `BSSID + 信道` 后发起连接
- `oled-screen` 额外使用兼容参数：
  - `WiFi TX Power = 8.5 dBm`
  - `WIFI_PS_NONE`
  - `PMF capable = false`
  - `11b/g/n + HT20`

当前这样处理的原因是：

- 无屏版本使用原逻辑即可稳定联网
- 带屏幕版本在默认连接参数下会出现认证阶段失败
- 实测验证后，带屏幕版本在“预扫描锁定 AP + 降低发射功率”策略下已经可以成功获取 IP

### 3.4 业务应用层

职责：

- 采样调度
- 数据组包
- 上报节奏控制
- 后续下行命令处理

当前对应模块可参考：

- `main/app/telemetry_app.c`
- `main/app_main.c`

### 3.5 调试与控制台层

职责：

- 串口日志输出
- 本地调试命令
- 给桌面 App 提供可解析的文本状态

当前对应模块可参考：

- `main/app/console_service.c`

## 4. 硬件接线与 IO 规划

### 4.1 当前已确认接线

- `DHT11 VCC -> 3V3`
- `DHT11 GND -> GND`
- `DHT11 DATA -> GPIO3`
- `板载 LED -> GPIO8`，低电平点亮

当前排查中已确认：

- `DHT11` 之前读不到数据的根因是接线错误，而不是固件逻辑错误
- 修正接线后，当前 `DHT11` 读取已经恢复正常
- 当前主固件不接 `OLED`，并且主流程中已屏蔽 OLED 依赖

### 4.2 OLED 预留接线

当前主固件没有启用 OLED，但若后续恢复显示，建议使用：

- `OLED SDA -> GPIO6`
- `OLED SCL -> GPIO7`

这样可以把 `GPIO4` 留给 I2C 的 `SDA`。

### 4.3 新增传感器推荐 IO

- 单总线传感器优先使用空闲 GPIO，例如 `GPIO5`
- I2C 传感器统一复用 `GPIO4 + GPIO5`
- ADC 传感器优先使用 `GPIO0 ~ GPIO2`

## 5. 推荐模块关系

```text
传感器驱动
  -> 业务应用层
  -> 设备数据模型
  -> 网络服务层(MQTT发布)
  -> 服务端/网页

系统状态
  -> 控制台日志
  -> 串口监控 App
```

## 6. 主要数据流

### 5.1 上行数据流

1. 传感器采样。
2. 固件将数据整理为 JSON。
3. 通过 MQTT 发布到服务器 Topic。
4. 服务端订阅后更新网页与历史记录。

### 5.2 本地调试数据流

1. Wi-Fi、MQTT、采样、发布结果输出到串口。
2. 桌面 App 监听串口日志。
3. App 解析日志并展示连接状态、传感器值和最近一次载荷。

## 7. 当前推荐输出内容

固件至少应输出以下调试信息：

- 启动完成
- Wi-Fi 连接成功/失败
- IP 地址
- MQTT 连接成功/失败
- 当前配置的传感器数量
- 当前成功读取的传感器数量
- 传感器读取结果
- 传感器读取失败原因或失败提示
- 发布成功/失败
- 等待条件不足时的提示

推荐兼容当前现有日志格式：

- `WiFi connected, ssid=... ip=...`
- `WiFi disconnected, reason=...`
- `DHT11 data pin: GPIO3`
- `MQTT connected`
- `MQTT disconnected`
- `DHT11 sample: temperature=... humidity=...`
- `sensor config: total=3 active=1`
- `sensor read failed: type=bh1750 reason=not found`
- `publish ok: {...}`
- `waiting dht11/wifi/mqtt...`

同时，当前固件已经开始提供结构化串口事件，便于桌面 App 不再只依赖人类可读日志：

- `APP_STATUS:{...}`
- `APP_CONFIG:{...}`
- `APP_OPTIONS:{...}`
- `APP_WIFI_LIST:[...]`
- `APP_EVENT:{"type":"wifi","data":{...}}`
- `APP_EVENT:{"type":"mqtt","data":{...}}`
- `APP_EVENT:{"type":"sensor","data":{...}}`
- `APP_EVENT:{"type":"publish","data":{...}}`
- `APP_EVENT:{"type":"runtime","data":{...}}`

这些日志当前主要用于：

- 对照传感器本地读数
- 判断 Wi-Fi、MQTT 还是传感器本身出现故障
- 供桌面 App 解析和展示设备运行状态

## 8. 后续演进建议

### 8.1 当前已完成的演进项

本节中，以下建议已经在当前主固件中落地：

1. 已将“串口人类可读日志”和“结构化串口协议”区分开。
   当前仍保留 `ESP_LOGI/W` 风格日志，供串口直接查看；同时新增 `APP_EVENT` / `APP_STATUS` / `APP_CONFIG` 等结构化输出，供桌面 App 稳定解析。
2. 已将固件版本号纳入统一配置与状态输出。
   当前通过 `APP_FIRMWARE_VERSION` 暴露固件版本，并写入设备状态和 MQTT 上报内容；版本格式统一为两段数字，例如 `1.0`。
3. 已将 MQTT 上报格式升级为统一 JSON。
   当前主消息已包含 `device`、`alias`、`ts`、`fwVersion`、`rssi` 和 `sensors` 分组结构。
4. 已把当前实际传感器能力纳入结构化上报。
   当前 `DHT11` 数据已按 `sensors.dht11.temperature` / `sensors.dht11.humidity` 输出，而不是继续使用散乱平铺字段。
5. 已将设备身份规范化。
6. 已增加硬件版本识别。
   当前通过探测板载 `OLED` 是否存在，区分 `supermini` 与 `oled-screen`，而不是依赖芯片型号判断开发板版本。
7. 已增加带屏幕板专用 Wi-Fi 连接策略。
   当前屏幕板会先扫描并锁定目标 AP 的 `BSSID` 与 `channel`，同时将发射功率降到 `8.5 dBm`，解决了该板在默认参数下反复认证失败的问题。
   当前设备真实上报到服务器的主身份字段为 `device`，格式为英文位置加序号，例如 `yard-01`；显示名字段为 `alias`，例如 `庭院 1 号`。
6. 已将 Wi-Fi 列表改为可由 App 下发并写入 NVS。
   当前设备名和传感器列表也支持通过 App 下发并持久化，设备重启后仍可恢复。

### 8.2 当前仍待继续推进的项

以下建议仍属于后续迭代方向，当前主固件尚未完整实现：

1. 将 Wi-Fi、MQTT 等运行参数完全从编译期宏迁移为“App 下发 + NVS 持久化”模式。
   当前 Wi-Fi 列表已支持通过 App 下发并写入 NVS，但 MQTT 地址、账号、Topic 仍主要来自编译期配置。
   当前编译期实际使用值为 `mqtt://117.72.55.63:1884`，账号和密码为空。
2. 为下行控制增加 MQTT Topic 订阅与超时保护。
   当前主流程仍以上报为主，尚未建立命令订阅、命令 ACK、执行超时和幂等保护机制。
3. 将更多传感器接入统一插件化框架。
   当前业务链路已经按多传感器数据结构预留，但真正落地的仍只有 `DHT11`。
4. 增加 OTA 升级链路。
   当前分区已经为单应用大分区模式，尚未切换到 OTA 双分区策略。
5. 将“传感器数量”和“传感器型号列表”直接纳入固件结构化状态。
   当前串口日志会输出 `sensor config: total=1 active=1`，桌面 App 主要依据本地配置推导当前传感器数量和型号，固件尚未在 `APP_STATUS` 中给出更完整的结构化多传感器状态。
6. 将 Topic 与设备类型的绑定从编译期常量改为配置化。
   当前主链路已从“固定传感器 Topic”切到“按设备 Topic 发布”，但 Topic 前缀仍来自编译期配置；后续仍建议迁移到 App 下发 + NVS 持久化。

### 8.3 MQTT 消息结构与服务端展示约定

当前主固件已经采用统一 JSON 结构向 MQTT 上报，但需要明确区分“当前真实实现”与“后续目标”，避免后续按过度理想化协议继续开发。

当前固件真实上报字段如下：

- `device`：设备唯一标识，当前为英文位置加序号，例如 `yard-01`。
- `alias`：设备显示名称，当前用于服务端和网页的中文展示，例如 `庭院 1 号`。
- `ts`：采样或上报时间戳，当前使用毫秒级 Unix 时间戳。
- `fwVersion`：固件版本号，当前固定为两段数字格式，例如 `1.0`。
- `rssi`：当前 Wi-Fi 信号强度。
- `sensors`：按传感器类型分组的数据对象。

当前固件真实上报示例：

```json
{
  "device": "study-01",
  "alias": "书房 1 号",
  "ts": 1711000000000,
  "fwVersion": "1.0",
  "rssi": -48,
  "sensors": {
    "dht11": {
      "temperature": 26.3,
      "humidity": 61.5
    }
  }
}
```

当前主 Topic 约定同步更新为：

- `device/<deviceId>`
- 示例：`device/study-01`

也就是说：

1. 每台设备只有一个主上报 Topic。
2. 设备上报到哪个 Topic，由 `deviceId` 决定，而不是由某一个固定传感器类型决定。
3. 一条消息里可以同时携带多个传感器，服务端应遍历 `payload.sensors` 中的每个传感器对象，而不是再通过 Topic 推断“这条消息里只能有一种传感器”。

当前要特别说明的是：

1. 当前固件已经不再上报 `source` 字段。
   服务端内部仍可能保留名为 `source` 的存储字段，但对来自 C3 的 MQTT 消息来说，当前应以 `device` 作为主识别键。
2. 当前固件已经不再上报 `ip` 字段。
   IP 更适合通过串口状态或本地运行态查看，而不是作为服务器长期存储的核心遥测字段。
3. 当前固件虽然使用了 `sensors` 分组结构，但真实接入并稳定工作的仍只有 `dht11`。
4. 当前 `device` 与 `alias` 的来源不是运行时动态自由输入，而是由设备名映射得到。
   例如 `庭院1号 -> yard-01 / 庭院 1 号`。
5. 当前 MQTT 连接入口已经修正为原生 TCP Broker。
   当前主固件实际连接的是 `mqtt://117.72.55.63:1884`，不是旧的 `ws://117.72.55.63/mqtt`。
6. 当前主固件已经不再使用 `garden/flower/dht11` 这类“按传感器固定主题”的发布方式。
   当前真实主规则为“每设备一个 Topic”，例如 `device/yard-01`、`device/study-01`。

当前服务端对这类消息的真实处理约定应理解为：

1. 服务端优先从 Topic 中拿到设备键，例如从 `device/study-01` 拿到 `study-01`。
2. 服务端同时读取 payload 中的 `device` 作为一致性补充，但主路由不再依赖“固定传感器 Topic”。
3. 服务端遍历 `sensors` 中的每个传感器对象，再把 `dht11`、`ds18b20`、`bmp280` 等真实传感器类型映射到网页内部使用的传感器槽位。
4. 网页侧设备页先按 `device` 归档，再显示该设备下实际存在的传感器数据。

因此，当前网页展示应按“当前已实现状态”来理解：

- 当前主链路已经支持 `dht11` 温湿度完整展示。
- 服务器代码已经预留 `ds18b20`、`bmp280`、`bh1750` 等传感器类型入口。
- 当前 C3 主固件已经按设备 Topic 发布，例如 `device/study-01`。
- 因此当前网页“真正可验证的主展示链路”已经调整为 `设备 Topic -> payload.sensors -> 设备页对应卡片`。

这一节对应的当前结论应明确为：

1. 固件当前已经把“设备身份”和“显示名称”规范化为 `device + alias`。
2. 固件当前已经把单传感器数据组织进 `sensors` 结构，为后续多传感器扩展打好了协议基础。
3. 服务端当前已经能正确接收并解析 C3 的真实 MQTT 上报，不再依赖旧的平铺 `temperature / humidity` 作为唯一入口。
4. 当前系统已经从“按传感器 Topic 路由”切到“按设备 Topic 路由”，链路更适合后续多传感器扩展。
5. 但当前网页仍保留固定设备目录和固定传感器槽位，尚未完全变成任意设备、任意传感器的全动态页面。

因此，后续修改应遵循以下原则：

1. 继续以 `device` 作为服务器侧的唯一主设备键。
2. `alias` 仅作为展示名，不再承担设备唯一识别职责。
3. 新增传感器时，优先扩展 `sensors.<sensorType>` 结构，不再回退到顶层平铺字段。
4. 在服务端和网页真正完成多传感器动态化之前，文档和实现都要明确写清“当前实际生效的是哪条链路”，避免把“预留能力”误写成“已经完全实现”。

### 8.4 中长期演进方向

后续固件可以从“当前固定参数烧录”逐步演进为“通用固件 + 配置下发”模式：

1. 同一份固件可烧录到多台 `ESP32-C3`。
2. 设备首次上电后由 App 配置设备名、Wi-Fi 和 MQTT 参数。
3. 配置写入 NVS 后，设备再进入正常联网与上报流程。
4. 传感器驱动按插件化方式扩展，而不是为每块板子维护一套独立固件。
5. 当多传感器真实接入后，再推动网页从“固定页面和固定卡片”演进为“按设备能力动态生成卡片”。
6. 当 OTA 与下行控制链路成熟后，再把当前单应用分区模式切换为更完整的量产型部署模式。
