# 网页架构

## 1. 目标定位

网页端是整个系统的可视化入口，负责把 `ESP32-C3` 上报的数据展示给用户，并提供后续可扩展的控制入口。

当前仓库中已经存在可参考实现：

- `03-server/01-net-display-server`

当前网页系统本质上是“服务端 + 前端页面”的组合，而不是纯静态网页。

## 2. 网页侧需求

### 2.1 展示需求

1. 展示当前最新的温湿度等传感器数据。
2. 展示设备在线状态、最后更新时间、数据来源。
3. 展示历史趋势曲线。
4. 后续支持多传感器、多设备分组展示。

### 2.2 交互需求

1. 页面自动刷新最新状态。
2. 支持切换不同时间范围查看历史数据。
3. 后续支持控制类操作，例如水泵控制、刷新设备状态。

### 2.3 服务端需求

1. 接收 MQTT 消息并更新内存态/数据库。
2. 提供 HTTP API 给前端读取数据。
3. 保存历史数据，支持曲线查询。
4. 对外提供公网访问入口或反向代理接入能力。

## 3. 推荐网页系统架构

### 3.1 服务端

建议继续采用 `Node.js` 作为网关服务，职责包括：

- 承接 MQTT 数据
- 提供 HTTP API
- 管理历史数据
- 提供静态页面资源

当前可参考文件：

- `03-server/01-net-display-server/server.js`

### 3.2 前端页面

前端负责：

- 拉取最新数据
- 展示状态卡片
- 绘制历史图表
- 后续扩展控制面板

当前可参考文件：

- `03-server/01-net-display-server/public/index.html`
- `03-server/01-net-display-server/public/app.js`

### 3.3 数据存储

建议服务端维护两类数据：

1. 最新状态缓存
2. 历史数据存储

当前项目已经有 SQLite 历史库能力，适合继续沿用。

### 3.4 实时推送演进方向

当前前端以 HTTP 拉取为主，后续可增加 `WebSocket / Socket.IO` 实时推送层，用于降低轮询压力并提升多设备刷新体验。

## 4. 推荐分层

### 4.1 接入层

职责：

- 订阅设备级 MQTT Topic
- 先按 Topic 中的 `deviceId` 识别设备
- 再解析 `payload.sensors` 中的具体传感器 JSON
- 做基础校验

### 4.2 业务层

职责：

- 设备状态归档
- 最新值更新
- 历史数据落库
- 设备在线离线判定

### 4.3 接口层

职责：

- 提供 REST API
- 给前端返回最新值、历史数据、设备状态

### 4.4 展示层

职责：

- 页面布局
- 图表与状态卡片
- 用户交互逻辑

## 5. 主要数据流

```text
ESP32-C3
  -> MQTT Broker
  -> Node.js 服务端
  -> 设备路由 + 传感器解析 + 缓存/历史库
  -> HTTP API
  -> 浏览器页面
```

## 6. 建议页面模块

1. 设备总览区
2. 传感器实时数据区
3. 历史曲线区
4. 设备在线状态区
5. 后续控制区
6. 设备管理区

设备管理区后续可承担：

- 设备显示名管理
- 设备分组或房间归类
- 稳定 `device` 标识与页面展示名映射

## 7. 当前适合保留的接口方向

根据现有仓库，建议保留以下能力：

- `GET /api/sensor/latest`
- `GET /api/sensor/history`
- `GET /api/mqtt/status`
- `GET /api/devices/status`

后续若需要控制类功能，可增加：

- `POST /api/device/command`
- `POST /api/pump/pulse`

## 8. 后续演进建议（已完成项标注状态）

1. ✅ **前端展示模型统一命名** — 前端 `SENSOR_ONLINE_WINDOW_MS` 与服务端 `DEVICE_ONLINE_WINDOW_MS` 均为 90 秒，已在代码注释中对齐说明。API 字段命名（`deviceSensors`、`points`、`ipAddresses` 等）与前端引用完全一致，无不一致字段。

2. ✅ **MQTT 接入和 HTTP API 逻辑分模块拆分** — 天气 API 逻辑已从 `server.js` 提取至 `lib/weather.js`，包含：URL 常量、缓存管理、`getWeatherDescription()`、`getWeatherForecast()`、`invalidateWeatherCache()`。`server.js` 通过 `require('./lib/weather')` 引入，主文件耦合度降低。

3. ✅ **补充设备告警、离线提醒、异常标记** — 已实现：
   - `ALERT_THRESHOLDS` 定义各传感器告警阈值（花花温度 >35°C 警告 / >40°C 异常，湿度 <20% / >90% 等）
   - `getMetricAlertLevel()` 函数返回 `"alert"` / `"warn"` / `null`
   - 设备卡片：指标数值颜色编码（橙色警告 / 红色异常），卡片标题旁显示告警徽章
   - 传感器详情页：指标数值和标签旁均显示颜色编码和徽章
   - 离线状态：`is-offline` CSS 类已有灰色状态点，`is-alert` 为红色（已存在样式）
   - CSS 新增：`.alert-badge`、`.alert-badge-warn`、`.alert-badge-alert`、`.metric-value-warn`、`.metric-value-alert`

4. ⏭️ **如果后续页面越来越复杂，可拆成独立前端工程** — 当前规模不需要，保留为未来方向。

5. ✅ **统一使用稳定英文标识** — 所有设备 ID（`yard-01`、`study-01`、`office-01`、`bedroom-01`、`server`、`weather`）及传感器 Key（`dht11`、`ds18b20`、`bmp280`、`bh1750`）均为英文，中文名称仅作 `title`/`subtitle` 展示字段。MQTT 主题、API 路径、数据库字段全部使用英文标识。

6. ✅ **MQTT 主路由已调整为按设备 Topic** — 当前服务端以 `device/<deviceId>` 作为 MQTT 主路由，先识别设备，再遍历 `payload.sensors` 解析具体传感器；网页展示仍沿用设备页和传感器卡片，但底层不再依赖“一个传感器一个固定 Topic”的旧模式。
