# 通信协议说明

## 1. 目标

本文件统一描述当前系统中与 `ESP32-C3` 相关的两类通信：

1. `C3` 和网络系统之间的通信
2. `C3` 和 App 之间的通信

这样后续无论是改固件、改服务端还是改上位机，都可以以这一份为统一依据。

## 2. C3 和网络的通信

### 2.1 通信角色

- `ESP32-C3`：数据采集节点
- `MQTT Broker / 网络入口`：消息传输层
- `服务端`：数据消费、展示和存储
- `网页前端`：读取服务端接口进行展示

### 2.2 当前推荐协议

当前仓库最贴合的方案是：

- `Wi-Fi` 负责联网
- `MQTT` 负责上报数据
- 消息体使用 `UTF-8 JSON`

当前主固件的上报节奏为：

- 周期发布
- 默认采样/发布间隔为 `3000 ms`
- 当前工程以上行上报为主，未启用额外 MQTT 订阅

### 2.3 上行通信内容

当前 C3 上报的重点不应是固定的温湿度字段，而应是“设备是谁、当前固件版本、这次成功采集到了几个传感器、每个传感器是什么类型以及对应数值”。

当前建议至少上报以下字段：

- `device`：设备唯一标识，命名上应能体现安装位置和序号，例如 `yard-01`、`balcony-02`。
- `source`：数据来源或板卡类型，例如 `esp32-c3`。
- `ts`：本次数据采集或上报时间戳。
- `fwVersion`：当前固件版本。
- `sensorCount`：本次成功采集并参与上报的传感器数量。
- `sensors`：传感器列表或传感器对象，内部包含传感器类型枚举和对应数据值。

这里特别约定：

1. `sensorCount` 表示“本次成功上报的传感器个数”，不是配置页面里理论上配置了几个。
2. App 或配置工具可以修改传感器配置数量，但 C3 只有在实际成功读取到传感器数据时，才允许把该传感器写入本次 MQTT 消息。
3. 如果配置了 3 个传感器，但当前只成功读取到 1 个，则本次消息中只上报 1 个传感器，同时 `sensorCount = 1`。
4. 因此服务端应以 `sensors` 实际内容为准，而不是假设某设备一定具备固定种类或固定数量的传感器。

推荐消息格式：

```json
{
  "device": "yard-01",
  "source": "esp32-c3",
  "ts": 1711000000000,
  "fwVersion": "1.0.0",
  "sensorCount": 2,
  "sensors": [
    {
      "type": "dht11",
      "values": {
        "temperature": 25.0,
        "humidity": 60.0
      }
    },
    {
      "type": "bh1750",
      "values": {
        "lux": 328
      }
    }
  ]
}
```

其中：

- `type` 用于表示传感器种类枚举，例如 `dht11`、`ds18b20`、`bmp280`、`bh1750`。
- `values` 用于承载该传感器类型对应的具体数值。
- 不要求所有设备都上报相同的传感器类型。
- 未成功读取到的数据不得伪造空值参与上报。

当前主固件里已经实际使用过的简化消息格式为：

```json
{
  "device": "yardHub",
  "alias": "庭院1号设备",
  "source": "yard-1-flower-c3",
  "temperature": 25.0,
  "humidity": 60.0,
  "rssi": -48,
  "ip": "192.168.1.135"
}
```

这个格式适合当前单一 `DHT11` 节点接入，但长期建议过渡到统一的 `sensorCount + sensors` 结构，便于多传感器扩展。

### 2.4 Topic 约定

当前可沿用仓库里的主题设计：

- 上报主题：`garden/flower/dht11`

当前主题的业务含义：

- 设备分组：庭院 1 号设备
- 数据类型：花花环境温湿度

后续可扩展：

- `garden/fish/ds18b20`
- `garden/climate/bmp280`
- `garden/light/bh1750`
- `garden/yard/pump/set`

中长期如果要过渡到“多设备统一平台”，建议逐步统一为更规范的层级，例如：

- `sensor/{device}/{sensorType}`
- `device/{device}/status`
- `device/{device}/config`
- `device/{device}/ota`

### 2.5 通信流程

```text
ESP32-C3
  -> 连接 Wi-Fi
  -> 连接 MQTT
  -> 发布传感器 JSON
  -> 服务端订阅并处理
  -> 网页通过 API 展示
```

### 2.6 网络协议要求

1. JSON 字段命名应保持稳定。
2. 数值字段必须传数字，不应传字符串。
3. `device` 应保持长期稳定，不能频繁变化。
4. `sensorCount` 必须与本次消息中的实际传感器数量一致。
5. 未成功读取的传感器不得写入 `sensors`。
6. 推荐后续增加下行订阅能力，用于控制与心跳。
7. 推荐服务端内部使用稳定英文 `device` 标识，中文名称单独放在 `alias`。

### 2.7 后续下行方向

未来建议补充两类网络下行通信：

1. 网关心跳状态下发
2. 设备控制命令下发

例如：

- `garden/gateway/status/<device>`
- `garden/gateway/broadcast`
- `garden/yard/pump/set`

## 3. C3 和 App 的通信

### 3.1 当前通信方式

当前仓库里 App 与 C3 的通信方式是：

- `USB Serial / UART`
- 数据格式为“文本日志流”

也就是说，App 目前主要是读取 C3 输出，而不是复杂双向协议。

### 3.2 当前可解析日志

当前固件建议稳定输出以下日志：

- `WiFi connected, ip=...`
- `WiFi disconnected, reason=...`
- `DHT11 data pin: GPIO4`
- `MQTT connected`
- `MQTT disconnected`
- `DHT11 sample: temperature=... humidity=...`
- `publish ok: {...}`
- `waiting dht11/wifi/mqtt...`
- `DHT11 read failed: ...`

### 3.3 App 侧处理方式

App 通过串口接收日志后，需要提取以下结构化信息：

- 是否已连接串口
- Wi-Fi 是否在线
- IP 地址
- MQTT 是否在线
- 最近一次传感器值
- 最近一次上报载荷

### 3.4 当前协议特点

优点：

- 实现简单
- 与现有固件兼容
- 调试成本低

限制：

- 协议不够严格
- 对日志文案变化敏感
- 双向交互能力较弱

## 4. C3 和 App 的后续建议协议

为提高稳定性，建议在保留现有日志的同时，增加结构化串口协议。

### 4.1 推荐做法

1. 继续输出人类可读日志。
2. 额外输出固定前缀的结构化状态行。
3. App 优先解析结构化状态行。

### 4.2 推荐结构化状态示例

```text
@state {"wifi":true,"ip":"192.168.1.135","mqtt":true,"temperature":25.0,"humidity":60.0}
@publish {"topic":"garden/flower/dht11","ok":true}
```

### 4.3 后续可扩展双向命令

如果后续需要 App 主动控制 C3，建议增加串口命令协议，例如：

```text
@cmd restart
@cmd wifi_reconnect
@cmd mqtt_reconnect
@cmd get_status
```

对应返回：

```text
@resp {"cmd":"get_status","ok":true,"wifi":true,"mqtt":true}
```

## 5. 统一约束建议

1. C3 和网络通信统一使用 JSON。
2. C3 和 App 通信建议逐步从“纯日志”升级到“日志 + 结构化协议”。
3. 所有协议都应带稳定字段名，避免同义字段混用。
4. 所有设备都应有唯一 `device` 标识。
5. 后续所有新增协议都应先更新本文档，再进入实现阶段。
