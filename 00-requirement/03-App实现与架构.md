# App 实现与架构

## 1. 目标定位

这里的 App 指当前仓库中的桌面上位机，用于通过串口连接 `ESP32-C3`，实时查看设备状态与日志。

当前仓库中已经存在可参考实现：

- `02-app/01-c3-monitor`

当前方案不是手机 App，而是本地桌面监控工具。

## 2. App 需求

### 2.1 基础需求

1. 能够选择串口并连接开发板。
2. 能持续接收串口日志。
3. 能解析固件输出中的关键状态。
4. 能将状态以可视化界面展示出来。
5. 能显示完整原始日志，便于排查问题。

### 2.2 解析需求

App 需要能解析至少以下信息：

- 串口连接状态
- 板子是否完成启动
- Wi-Fi 是否连接
- 当前 IP 地址
- MQTT 是否连接
- 最近一次温湿度
- 最近一次上报 JSON

### 2.3 后续扩展需求

1. 增加串口命令发送能力。
2. 增加日志导出能力。
3. 增加设备参数配置能力。
4. 增加多设备切换能力。

## 3. 当前推荐实现方式

### 3.1 技术栈

建议沿用当前栈：

- `Python`
- `PySide6 / Qt`
- `pyserial`

当前目录可参考：

- `02-app/01-c3-monitor/main.py`
- `02-app/01-c3-monitor/app/window.py`
- `02-app/01-c3-monitor/app/serial_worker.py`
- `02-app/01-c3-monitor/app/parser.py`

### 3.2 模块划分

#### 界面层

职责：

- 串口选择与连接按钮
- 状态面板展示
- 原始日志展示

对应文件：

- `app/window.py`

#### 串口通信层

职责：

- 打开/关闭串口
- 持续读取数据
- 推送给界面线程

对应文件：

- `app/serial_worker.py`

#### 日志解析层

职责：

- 解析 Wi-Fi、MQTT、传感器日志
- 从文本中提取结构化状态

对应文件：

- `app/parser.py`

## 4. 推荐架构关系

```text
ESP32-C3 串口日志
  -> Serial Worker
  -> Log Parser
  -> UI State
  -> Qt 窗口展示
```

## 5. 当前解析策略

当前 App 推荐继续兼容“文本日志解析”模式，不强依赖固件新增协议。

优点：

- 不需要立刻改固件
- 能直接接入现有项目
- 对调试阶段很友好

限制：

- 依赖日志文案稳定
- 不如结构化协议可靠

因此建议后续逐步演进为：

1. 保留人类可读日志。
2. 额外增加结构化串口状态行。
3. App 优先解析结构化协议，日志区继续显示原文。

## 6. App 页面建议

1. 顶部连接区
2. 设备状态区
3. Wi-Fi / MQTT 状态区
4. 传感器实时值区
5. 最近上报载荷区
6. 原始日志区

## 7. 后续演进建议

1. 如果未来要做手机 App，建议把“通信协议”和“UI 逻辑”从当前桌面 App 设计中抽离。
2. 可增加“发送命令”面板，用于重启、重连、查询配置。
3. 可增加固件版本、设备标识、最后心跳时间显示。
4. 可新增日志级别过滤和关键词搜索。

## 8. 中长期 App 方向

如果后续项目需要“同一份固件批量部署”，App 可以进一步分成两类：

### 8.1 桌面调试 App

继续承担：

- 串口监控
- 日志解析
- 本地调试

### 8.2 手机配置 App

未来若增加 BLE 配网，可新增手机 App，负责：

- 扫描待配置设备
- 写入设备名
- 写入 Wi-Fi 参数
- 写入 MQTT 地址和采集配置

这样可以把“调试工具”和“交付使用时的配置工具”区分开，避免当前桌面串口工具承担过多职责。
